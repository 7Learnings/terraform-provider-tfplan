# Terraform Provider `tfplan`

This provider introduces the `tfplan` resource, which allows you to read the outputs from an upstream Terraform stack's plan file (`tfplan.json`). This enables powerful cross-stack dependency and planning capabilities that are not possible with Terraform's native `terraform_remote_state` data source.

## The `tfplan` Resource

The core of this provider is the `tfplan` resource. It is configured with a path to an upstream Terraform stack.

Example usage:

```hcl
resource "tfplan" "upstream" {
  path = "../my-other-stack"
}

output "upstream_output_value" {
  value = tfplan.upstream.outputs["some_output_key"]
}
```

## Cross-Stack Planning

The `tfplan` resource provides a unique capability for cross-stack planning by reading the plan file of an upstream stack, rather than its state file.

### How it Works

During the `terraform plan` phase, the `tfplan` provider reads the `tfplan.json` file generated by the upstream stack. This file contains the *planned values* for the upstream stack's outputs. This means a downstream stack can see what the outputs of an upstream stack *will be* after an apply, before the upstream stack is actually applied.

Outputs that are known at plan time in the upstream stack will be available as known values in the downstream stack's plan. Outputs that are not known until apply time (e.g., depending on a resource that hasn't been created yet) will be marked as "known after apply" in the downstream plan, correctly propagating the dependency.

After a `terraform apply`, the provider will read the final output values from the `outputs.json` file in the upstream stack directory to ensure the state is consistent.

### Comparison to `terraform_remote_state`

This approach is fundamentally different from the `terraform_remote_state` data source:

-   `terraform_remote_state`: Reads from a remote state file, which reflects the last *applied* state of a stack. It has no knowledge of pending changes in a new plan. This means you can only depend on what is already deployed.
-   `tfplan` resource: Reads from a plan file (`tfplan.json`). This allows a downstream stack to depend on *planned changes* in an upstream stack, enabling true cross-stack dependency planning. You can plan a whole series of dependent stacks together.

This makes it possible to orchestrate complex deployments across multiple stacks with a single, unified planning step.

## Design Philosophy: A Deliberate Contract

The `tfplan` provider is intentionally designed to only expose the `outputs` of an upstream stack. It does not provide access to arbitrary resource attributes within that stack.

This design choice is deliberate:

-   **Explicit Contract**: A Terraform module's or stack's `outputs` are its public API or contract. They represent the values that the author has chosen to expose for consumption.
-   **Implementation Details**: Resource attributes, on the other hand, are implementation details of a stack. They are subject to change, refactoring, or removal without notice. Relying on them would create brittle dependencies between stacks.

By limiting access to only the defined `outputs`, the `tfplan` provider ensures that dependencies between your stacks are based on a stable, explicit contract, leading to a more robust and maintainable infrastructure-as-code setup.
